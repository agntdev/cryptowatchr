# fix-36211a5b111a0840 — PostgreSQL sent_alerts table accumulates stale cooldown records with no periodic cleanup

**Weight:** 0.0000 (share of project budget)
**Reward:** 0 CWTR

`PostgresStore.recordSentAlert()` (`src/store.ts:752-758`) upserts rows into the `sent_alerts` table with an `expires_at` timestamp. Expired records are only cleaned lazily inside `isAlertSuppressed()` (line 761-776) — and only when that specific `(user_id, rule_id)` pair is queried again. If a rule is deleted, or the user becomes inactive, the stale cooldown row persists forever.

In contrast, `RedisStore.recordSentAlert()` (line 214-217) uses the Redis `PX` (expiry) option, so stale keys auto-evict. The Postgres implementation has no equivalent periodic expiry job — no `DELETE FROM sent_alerts WHERE expires_at < $1` is ever scheduled.

Over the bot's lifetime this table will grow without bound, progressively degrading `isAlertSuppressed` query performance.
