# fix-c599a358a92904a5 — `/start` bypasses onboarding — shows main menu instead of timezone selection (spec break for E1T1/E1T2/E1T3)

**Weight:** 0.0000 (share of project budget)
**Reward:** 0 CWTR

The `/start` handler at `src/bot.ts:706-713` sends `MAIN_MENU_TEXT` and a `mainMenu()` keyboard, which skips the entire onboarding flow. The E1T1 spec says `/start` must "explain the bot's features and initiate the onboarding flow," and the E1T1.json test expects `WELCOME_TEXT` with `timezoneKeyboard()` on the very first `/start` (lines 7-32 of `tests/specs/E1T1.json`).

**Impact:** The onboarding infrastructure exists but is dead code — the `onboard:tz:*`, `onboard:qhours`, `onboard:qhours:skip`, `onboard:skip`, and `onboard:done` callback handlers are never reachable from a normal user interaction. Likewise, the `onboardingStep === "timezone"` and `quietHoursStep` message handlers (which handle free-text timezone input and HH:MM quiet-hour answers during onboarding) are unreachable. The `Session.onboardingStep` field is only ever set to `undefined` (by `/start`) or `"confirm"` (by an unreachable callback) — the `"timezone"` value declared in the Session interface is never assigned at all.

The E1T1/E1T2/E1T3 specs all rely on this onboarding entrypoint. Because `/start` skips it, the E1T1 spec test for "selecting a timezone shows defaults and confirmation" and "typing a custom timezone during onboarding" will fail, and the E1T2/E1T3 onboarding-dependent flows are unreachable.

**Fix:** Change the `/start` handler to show `WELCOME_TEXT` with `timezoneKeyboard()` (and set `onboardingStep = "timezone"` in session) instead of jumping directly to the main menu. The existing confirm/callback logic is already wired correctly and just needs the correct trigger.
